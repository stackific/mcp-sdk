# OAuth.DiscoverOAuthMetadataAsync method

Discovers protected-resource metadata (RFC 9728) then authorization-server metadata (RFC 8414/OIDC) for an MCP endpoint, walking the full well-known fallback ordering and validating both documents (spec §23.2–§23.3).

```csharp
public static Task<OAuthMetadata> DiscoverOAuthMetadataAsync(HttpClient http, string resource, 
    string? resourceMetadataUrl)
```

| parameter | description |
| --- | --- |
| http | The HTTP client to use (injectable for offline tests). |
| resource | The canonical resource identifier (the protected MCP endpoint URL). |
| resourceMetadataUrl | The protected-resource metadata URL from the `401` challenge, or `null` to use the well-known fallback ordering. |

## Return Value

The discovered, validated metadata (issuer, protected-resource, authorization-server).

## Exceptions

| exception | condition |
| --- | --- |
| [McpError](../../Stackific.Mcp.JsonRpc/McpError.md) | When no valid, issuer-matching metadata can be discovered. |

## Remarks

Protected-resource discovery honours the `resource_metadata` precedence: when *resourceMetadataUrl* is supplied (typically from the `401`'s `WWW-Authenticate` header, R-23.2-d) it is used directly; otherwise the path-aware then host-root well-known URIs are tried in order, using the first that yields a valid document (R-23.2-e, R-23.2-f). The returned `resource` MUST equal *resource* (R-23.2-h).

The selected issuer's metadata is then discovered by trying the OAuth-AS-Metadata and OpenID-Connect-Discovery well-known URIs in the mandated priority order (R-23.3-c) and validating the document's `issuer` EXACTLY matches the issuer used to fetch it — the RFC 9207 mix-up defence (R-23.3-d, R-23.3-e). A document whose `issuer` differs is rejected.
