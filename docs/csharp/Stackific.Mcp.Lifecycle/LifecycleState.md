# LifecycleState enumeration

The three lifecycle states a governed protocol feature can be in (spec §27.1). Every governed feature is in exactly one of these states at any time, and the allowed transitions between them (enforced by `Stackific.Mcp.Lifecycle.LifecyclePolicy`) encode the deprecation policy of §27.2.

## Values

| name | value | description |
| --- | --- | --- |
| Active | `0` | Fully supported and recommended; implemented exactly as specified (R-27.1-a). Wire value `active`. |
| Deprecated | `1` | Still defined and functional, but discouraged for new use and scheduled for eventual removal; carries a migration note (R-27.1-b). Wire value `deprecated`. |
| Removed | `2` | Not defined by the document; carries no meaning and imposes no obligation. A Removed feature is simply absent from the spec text and registries (R-27.1). Wire value `removed`. |

## Remarks

Wire values are the lowercase strings `active`, `deprecated`, and `removed`, matching the TypeScript `LifecycleState` constant object exactly. The legal transitions are:

* Active → Deprecated — always permitted (R-27.2).
* Deprecated → Active — restoration permitted (R-27.2-n).
* Deprecated → Removed — permitted once the window elapses (R-27.2-a).
* Active → Removed — FORBIDDEN; a feature MUST pass through Deprecated first (R-27.2-b).
* Any transition out of Removed — FORBIDDEN; Removed is terminal (R-27.2).
