## Remarks

Every request is self-describing via its own `_meta`; servers MUST NOT infer identity, capabilities, or protocol version from any earlier request or from the underlying connection. Cross-request continuity rides on explicit, server-minted, opaque identifiers that the client echoes back verbatim. This module introduces no new wire types.

The normative stateless-processing rules every S06-conformant server must satisfy (enforced via the self-describing `_meta` envelope in [`RequestMeta`](./RequestMeta.md)): a server MUST NOT infer state from earlier requests, even on the same connection (R-4.4-a); MUST NOT require any prior request before processing a given one (R-4.4-b); MUST derive identity, capabilities, and version solely from the current `_meta` (R-4.4-c); MUST NOT depend on persisted per-connection conversational state (R-4.4-d); MUST NOT treat connection/process identity as a proxy for a conversation (R-4.4-f); cross-request state MUST be referenced by an explicit identifier rather than connection identity (R-4.5-a); and list results MUST NOT vary based on connection identity (R-4.6-a).

Three further stateless-model behaviors are RECOMMENDED (SHOULD) at the transport layer and cannot be enforced by this library — the HTTP / SSE transports satisfy them with transport-specific mechanisms: supporting interleaved task streams so unrelated requests on one connection do not head-of-line block (R-4.4-h); not requiring connection reuse between requests in the same logical conversation (R-4.4-i); and supporting mid-task resume on a new connection by accepting a continuation identifier from a prior connection's response (R-4.4-j).
