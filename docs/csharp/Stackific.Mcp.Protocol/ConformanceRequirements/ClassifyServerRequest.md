# ConformanceRequirements.ClassifyServerRequest method

Applies the ordered §29.2 baseline-server request checks to ONE self-contained §4 request and returns its [`ServerRequestDisposition`](../ConformanceRequirements.ServerRequestDisposition.md) (spec §29.2, R-29.2-e – R-29.2-n, R-29.4-k). The checks run strictly in the §7 flow order — judged on this request's own envelope, NEVER on connection or prior-request state (R-29.1-e, R-29.2-f):

```csharp
public static ServerRequestDisposition ClassifyServerRequest(JsonObject meta, 
    IReadOnlyList<string> serverSupportedRevisions, JsonObject? requiredClientCapabilities = null, 
    bool? featureAdvertised = null)
```

| parameter | description |
| --- | --- |
| meta | The request's `params._meta` envelope (raw). |
| serverSupportedRevisions | The revisions the server supports. |
| requiredClientCapabilities | The capabilities required to process this request (raw), or `null`. |
| featureAdvertised | Whether the feature is gated behind an advertised capability; `null` ⇒ unconstrained. |

## Return Value

The request disposition.

## Remarks

1. revision supported? → else `-32004` (data: supported, requested);
2. all §4-required fields present? → else `-32602` (Invalid params);
3. required client capability declared? → else `-32003` (data.requiredCapabilities);
4. feature gated by an advertised capability? → else refuse (not advertised);
5. else → accept.

A malformed (non-`YYYY-MM-DD`) protocol-version value is an envelope failure (`-32602`), not a `-32004`: the revision check first asks whether the declared revision is a well-formed, server-unsupported one.
